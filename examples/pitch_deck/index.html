Research Problem

Execution engines are strong at control-flow, weak at explainability.

Opaque Runtime State

Conventional logs capture events but not semantically linked state transitions and decision rationale.

Weak Interoperability

Process tools often expose proprietary diagnostics instead of reusable RDF vocabularies and query interfaces.

Limited Causal Analysis

Teams can answer "what happened" but struggle with "why did this branch fire and what alternatives were excluded?"

Approach

With SPEAR, every process instance is queryable at runtime.

SPEAR maps process definitions, instance state, tokens, variables, and audit events into RDF graphs. This enables direct SPARQL queries for traceability, compliance, and debugging workflows.

SPARQL View

Example analysis questions exposed as queries

Trace executed activities per instance

SELECT ?instance ?activity ?ts WHERE {
  ?event <http://example.org/audit/instance> ?instance ;
         <http://example.org/audit/activity> ?activity ;
         <http://example.org/audit/timestamp> ?ts .
}
ORDER BY ?instance ?ts

Find stalled instances with waiting tokens

SELECT ?instance ?node WHERE {
  ?token a <http://example.org/instance/Token> ;
         <http://example.org/instance/status> "WAITING" ;
         <http://example.org/instance/currentNode> ?node .
  ?instance <http://example.org/instance/hasToken> ?token .
}
Use Case 1

Agentic Customer Support with Human Escalation

Scenario: AI drafts responses, but policy-sensitive or high-value tickets require deterministic checks and supervisor sign-off.

Ticket Ingest Policy Check Task Knowledge API Calls Human Approval Gateway Response + Audit

Pilot target band: 8-18% lower first-response time and 30-60% less analyst time spent explaining escalated decisions.

Assumes current flow has manual policy triage and no unified decision provenance.

Use Case 2

Insurance Claims Adjudication in Regulated Environments

Scenario: Claims routing, fraud checks, and payout approvals need repeatable logic and clear evidence trails.

Claim Intake Fraud Scoring API Rules Gateway Adjuster Task Settlement + Audit Export

Pilot target band: 10-25% lower median cycle time and 40-70% reduction in audit evidence assembly effort.

Assumes today's evidence is spread across BPM logs, adjuster notes, and API traces.

Use Case 3

B2B Order-to-Cash Across Fragmented Systems

Scenario: ERP, inventory, credit, and logistics APIs often fail asynchronously and create invisible bottlenecks.

Order Received Inventory/Price Checks Credit Decision Gateway Fulfillment + Notifications Exception Handling

Pilot target band: 5-15% higher straight-through processing and 20-40% lower MTTR for cross-system order failures.

Assumes recurring retry/reconciliation issues between ERP, credit, and fulfillment services.

Pilot Plan

Start with one high-friction workflow and prove value in weeks.

Week 1

Model BPMN flow and define integration handlers.

Week 2

Run sandbox traffic with full audit and error-path coverage.

Week 3-4

Measure baseline vs pilot on cycle-time, exception MTTR, and explanation lead-time.

Reporting discipline: present median and p90, plus sample size and confidence intervals where available.

Evidence Protocol

How to present claims credibly at a semantic web conference

1. Matched Baseline

Capture 2-4 weeks of pre-pilot metrics for the same workflow, team mix, and ticket/order complexity profile.

2. Controlled Pilot

Run SPEAR on the same class of cases. Keep SLAs and staffing stable to isolate orchestration + semantic traceability effects.

3. Query-Backed Results

Publish SPARQL query snippets used to compute KPIs so audience can inspect definitions, not just final numbers.

Limitations

Current constraints and open technical questions

Not A Triplestore-First Runtime Yet

Current persistence is file-backed RDF in this repo; production triplestore deployment patterns are part of the next phase.

Semantic Profiles In Progress

PROV-O and SHACL alignment is planned and should be shown with concrete mappings, not only architecture claims.

Benchmarking Scope

Performance and reasoning overhead benchmarks are not yet fully published across heterogeneous workloads.
